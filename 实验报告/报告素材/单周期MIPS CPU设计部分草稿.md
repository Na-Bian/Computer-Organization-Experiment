# **CPU设计实验**

## **单周期MIPS CPU设计**

### **设计思路**

1.  总体设计思想

单周期 MIPS CPU 的基本特点是：每条指令均在一个时钟周期内完成取指、译码、执行、访存和写回等全部操作。因此，单周期处理器不需要为不同指令设置多个执行状态，而是通过一套完整的数据通路和硬布线控制器，在一个周期内同时给出各部件所需的控制信号。

本实验中设计的单周期 MIPS CPU 需要支持 `add`、`slt`、`addi`、`lw`、`sw`、`beq`、`bne`、`syscall` 共 8 条核心指令，并能够运行实验材料中的 `sort.hex` 冒泡排序程序。为满足该要求，CPU 总体上采用“指令存储器 ROM + 寄存器堆 + ALU + 数据存储器 RAM + 硬布线控制器”的结构。PC 给出当前指令地址，指令 ROM 输出机器指令；硬布线控制器根据指令中的 `OP` 字段和 R 型指令的 `Func` 字段产生控制信号；寄存器堆、ALU、数据存储器及各多路选择器根据控制信号完成对应的数据传送与运算。

单周期 CPU 的设计重点在于控制信号必须在同一周期内稳定产生，且不同指令的数据通路要能够共用同一套硬件资源。对于 R 型指令，ALU 两个输入均来自寄存器堆，结果写回 `rd`；对于 `addi` 指令，ALU 第二个输入来自符号扩展立即数，结果写回 `rt`；对于 `lw` 与 `sw` 指令，ALU 用于计算访存地址，数据存储器完成读写；对于 `beq` 与 `bne` 指令，ALU 通过减法比较两个寄存器是否相等，再由分支控制逻辑决定 PC 是否跳转；对于 `syscall` 指令，CPU 不再继续执行后续指令，而是产生停机信号。

2.  各种指令数据通路

单周期 CPU 中，各类指令虽然都在一个周期内完成，但实际经过的数据通路不同。根据本实验支持的 8 条指令，可以将数据通路归纳如下。

表2 单周期 MIPS 主要指令数据通路

| 指令类型 | 代表指令 | 主要数据通路 |
| --- | --- | --- |
| R 型运算指令 | `add`、`slt` | `PC -> 指令ROM -> 寄存器堆读rs/rt -> ALU运算 -> 写回rd` |
| 立即数运算指令 | `addi` | `PC -> 指令ROM -> 寄存器堆读rs -> ALU与SignExt(imm)相加 -> 写回rt` |
| 访存读指令 | `lw` | `PC -> 指令ROM -> 寄存器堆读rs -> ALU计算地址 -> RAM读数据 -> 写回rt` |
| 访存写指令 | `sw` | `PC -> 指令ROM -> 寄存器堆读rs/rt -> ALU计算地址 -> 将rt数据写入RAM` |
| 条件分支指令 | `beq`、`bne` | `PC -> 指令ROM -> 寄存器堆读rs/rt -> ALU作减法比较 -> 条件成立时更新PC` |
| 停机指令 | `syscall` | `PC -> 指令ROM -> 控制器识别Func=001100 -> 产生Halt信号` |

对于 `add` 和 `slt`，控制器令 `RegDst=1`，选择 `rd` 作为写寄存器编号；令 `RegWrite=1`，允许将 ALU 运算结果写回寄存器堆。对于 `addi` 和 `lw`，目标寄存器均为 `rt`，因此 `RegDst=0`。其中 `addi` 的写回数据来自 ALU，`lw` 的写回数据来自数据存储器，因此二者需要通过 `MemToReg` 区分写回数据来源。

对于 `sw`，寄存器堆读出的 `R1` 用作基地址，符号扩展后的立即数用作偏移量，两者经 ALU 相加得到数据存储器地址；寄存器堆读出的 `R2` 即 `rt` 的内容作为写入数据，送入数据存储器写数据端。该指令不写寄存器，因此 `RegWrite=0`，并且 `MemWrite=1`。

对于 `beq` 和 `bne`，ALU 对 `rs` 和 `rt` 执行减法，通过 Zero 标志判断二者是否相等。`beq` 在 Zero 为 1 时跳转，`bne` 在 Zero 为 0 时跳转。由于本实验框架中的指令存储器按字寻址，PC 每次顺序执行时加 1，分支目标地址按照实验电路的数据通路由 PC 与符号扩展立即数生成。

3.  单周期 MIPS控制器设计过程

单周期硬布线控制器的输入主要为指令的 `OP` 字段和 `Func` 字段，输出为数据通路所需的各类控制信号。设计时首先完成指令译码，再由译码结果组合生成控制信号。

指令译码部分首先根据 `OP` 字段判断指令类型：

表3 单周期硬布线控制器 OP 字段译码

| 指令 | OP |
| --- | --- |
| R 型指令 | `000000` |
| `lw` | `100011` |
| `sw` | `101011` |
| `beq` | `000100` |
| `bne` | `000101` |
| `addi` | `001000` |

对于 R 型指令，还需要继续根据 `Func` 字段判断具体操作：

表4 R 型指令 Func 字段译码

| 指令 | Func |
| --- | --- |
| `add` | `100000` |
| `slt` | `101010` |
| `syscall` | `001100` |

因此，控制器中的典型译码逻辑可表示为：

```text
LW      = (OP == 100011)
SW      = (OP == 101011)
BEQ     = (OP == 000100)
BNE     = (OP == 000101)
ADDI    = (OP == 001000)
R_TYPE  = (OP == 000000)

ADD     = R_TYPE AND (Func == 100000)
SLT     = R_TYPE AND (Func == 101010)
SysCall = R_TYPE AND (Func == 001100)
```

需要特别注意，`ADD`、`SLT`、`SysCall` 不能只判断 `Func` 字段，必须同时保证 `OP=000000`。否则某些 I 型指令的低 6 位如果刚好与某个 R 型指令的 `Func` 字段相同，就可能被误判为 R 型指令。

![图2 单周期硬布线控制器指令译码逻辑](图片素材/单周期MIPS——硬布线控制器/指令译码逻辑.png)

控制信号由译码信号组合得到。根据各指令的数据通路，主要控制信号设计如下。

表5 单周期硬布线控制器主要控制信号

| 控制信号 | 作用 | 设计逻辑 |
| --- | --- | --- |
| `RegDst` | 选择写寄存器编号来自 `rt` 还是 `rd` | `RegDst = ADD OR SLT` |
| `RegWrite` | 控制寄存器堆是否写入 | `RegWrite = ADD OR SLT OR ADDI OR LW` |
| `MemWrite` | 控制数据存储器是否写入 | `MemWrite = SW` |
| `MemToReg` | 选择写回寄存器的数据来源 | `MemToReg = LW` |
| `AluSrc` | 选择 ALU 第二操作数来自寄存器还是立即数 | `AluSrc = ADDI OR LW OR SW` |
| `BEQ` | 标记当前指令为 `beq` | `BEQ = (OP == 000100)` |
| `BNE` | 标记当前指令为 `bne` | `BNE = (OP == 000101)` |
| `Halt` | 停机控制 | `Halt = R_TYPE AND SysCall` |

其中，`RegWrite` 不能简单写成 `R_TYPE OR ADDI OR LW`，因为 `syscall` 也是 R 型指令，但它不应写寄存器。若将所有 R 型指令均视为可写回指令，就可能在停机时误写寄存器。最终采用 `RegWrite = ADD OR SLT OR ADDI OR LW`，只对真正需要写寄存器的指令打开写使能。

ALU 控制逻辑用于根据指令类型产生 ALU 的运算选择信号。`lw`、`sw`、`addi` 需要 ALU 执行加法；`beq`、`bne` 需要 ALU 执行减法；R 型指令则根据 `Func` 字段进一步决定执行加法或有符号小于比较。`syscall` 被识别后不作为普通 R 型运算处理，而是产生停机信号。

![图3 单周期硬布线控制器控制信号](图片素材/单周期MIPS——硬布线控制器/控制信号.png)

![图4 单周期硬布线控制器 ALU 控制逻辑](图片素材/单周期MIPS——硬布线控制器/ALU控制逻辑.png)

### **（2）电路图**

#### **（2.1）硬布线控制器电路图**

单周期硬布线控制器由指令译码逻辑、主控制信号生成逻辑和 ALU 控制逻辑三部分组成。指令译码逻辑根据 `OP` 和 `Func` 产生 `LW`、`SW`、`BEQ`、`BNE`、`ADDI`、`ADD`、`SLT`、`SysCall` 等中间信号；主控制逻辑根据这些中间信号产生 `RegDst`、`RegWrite`、`MemWrite`、`MemToReg`、`AluSrc`、`Halt` 等控制信号；ALU 控制逻辑进一步产生送入 ALU 的运算控制码。

在控制器设计中，`syscall` 作为特殊 R 型指令单独处理。它的 `OP` 字段为 `000000`，但不应被归入普通 R 型运算指令，因此设计中将 `ADD`、`SLT` 与 `SysCall` 分别由 `R_TYPE` 与不同 `Func` 条件共同确定。这样既能保证 `add`、`slt` 正常写回寄存器，也能保证 `syscall` 只产生停机信号，不触发寄存器写入或存储器写入。

#### **（2.2）CPU设计图**

单周期 CPU 顶层电路以 PC 为起点。PC 输出作为指令 ROM 地址，指令 ROM 输出当前指令。指令字段被拆分后，一方面送入寄存器堆作为读寄存器编号和写寄存器编号候选，另一方面送入硬布线控制器产生控制信号。寄存器堆读出的两个操作数分别送入 ALU 或数据存储器写数据端；立即数字段经过符号扩展后作为 ALU 第二操作数的候选输入。

ALU 的输出一方面可作为 R 型指令和 `addi` 指令的写回数据，另一方面可作为 `lw/sw` 指令访问数据存储器的地址。数据存储器的读出数据与 ALU 结果共同送入写回数据选择器，由 `MemToReg` 决定最终写回寄存器堆的数据。PC 更新逻辑则根据普通顺序执行地址和分支目标地址进行选择，在 `beq/bne` 条件满足时改变 PC，否则 PC 顺序前进。执行 `syscall` 时，控制器产生 `Halt` 信号，使 CPU 停止继续更新。

### **（3）测试图**

测试时，将实验提供的 `sort.hex` 加载到指令 ROM 中，复位后运行 CPU。程序首先通过 `addi` 指令初始化寄存器，再通过 `sw` 指令将待排序数据写入数据存储器，随后执行 `lw`、`slt`、`beq`、`bne`、`sw` 等指令完成冒泡排序，最终通过 `syscall` 停机。测试过程中重点观察 PC、IR、寄存器堆写入端口、数据存储器写入端口以及 RAM 中排序区域的数据变化。

首先测试 `addi $s0,$0,-1`。该指令要求从 `$zero` 读取 0，与符号扩展后的立即数 `0xffff` 相加，得到 `0xffffffff` 并写入 `$s0`。测试结果显示，寄存器堆中 `$s0` 得到正确写入，说明 `addi` 的符号扩展、ALU 加法、`RegDst=0` 选择 `rt`、`RegWrite` 写使能和写回路径均工作正常。

![图9 执行 addi 后寄存器堆的存储情况](图片素材/单周期MIPS——硬布线控制器/执行addi $s0,$0,-1后寄存器堆的存储情况.png)

随后测试 `sw $s0,512($s1)`。该指令要求以 `$s1` 为基地址，将立即数 512 作为偏移量，经 ALU 计算得到数据存储器地址，并把 `$s0` 中的数据写入对应内存单元。测试结果显示，主存中对应位置写入了期望数据，说明 `sw` 指令的地址计算、数据存储器写使能、写入数据来源和 RAM 访问通路正确。

![图10 执行 sw 后主存的存储情况](图片素材/单周期MIPS——硬布线控制器/执行sw $s0,512($s1)后主存的存储情况.png)

在完整程序测试中，CPU 能够继续执行排序程序，并在运行结束后通过 `syscall` 停机。数据存储器排序区域的数据顺序符合程序预期，说明单周期 CPU 对 8 条核心指令的支持基本正确。

### **（4）故障与测试分析**

#### **（4.1）控制信号异常**

故障现象1：在调试单周期硬布线控制器时，部分控制信号在接入后出现异常。尤其在测试 `OP=100011` 的 `lw` 指令和 `OP=001000` 的 `addi` 指令时，相关控制信号有时不能按照预期独立变化。最初判断可能是 `LW` 与 `ADDI` 等隧道信号命名冲突，或者某些隧道标签重复导致两个逻辑信号被错误合并。

原因分析：进一步检查电路后发现，真正原因不是隧道命名错误，而是在一个与门输入附近，两条本应相互独立的信号线被误连在了一起。由于 Logisim 中线缆交叉与连接点较为接近，如果在交叉处形成了连接节点，就会导致两个控制条件在进入门电路前已经短接。这样一来，当某条指令译码信号为 1 时，另一条无关控制信号也会被牵连，进而使控制逻辑输出异常。

解决方案：删除可疑门电路并检查其下方隐藏连线，发现内部确实存在误连。随后重新绘制该处线路，保证每个译码信号独立进入逻辑门，不应连接的交叉线之间不形成节点。修正后，`lw`、`addi` 等指令对应的控制信号恢复正常。

![图11 删除与门前的异常连接位置](图片素材/单周期MIPS——硬布线控制器/删掉与门之前.png)

![图12 删除与门后发现内部误连](图片素材/单周期MIPS——硬布线控制器/删掉与门之后，发现内部的线连在了一起.png)

#### **（4.2）sort.hex 加载位置错误**

故障现象2：实验初期本地看似已经加载了 `sort.hex`，但在自动测试或重新打开电路时，程序无法按照预期执行，表现为指令数据没有被正确识别。

原因分析：最初将 `sort.hex` 加载到了 RAM 中，但在 CPU 设计中，RAM 主要作为运行时数据存储器，供 `lw` 和 `sw` 指令访问。指令内容应存放在指令存储器中，由 PC 作为地址读取。如果将程序文件放在数据 RAM 中，CPU 取指阶段无法从正确位置读出机器指令，而且 RAM 的内容也不适合作为电路保存时固定的程序初始内容。

解决方案：将指令存储器改为 ROM，并将 `sort.hex` 加载到指令 ROM 中保存。这样 PC 输出地址后，可以通过 `PC -> 指令ROM -> 指令字段/控制器` 的路径稳定取指；数据 RAM 则仅用于程序运行过程中的数据读写，不再预先加载 `sort.hex`。

![图13 指令存储器必须使用 ROM](图片素材/单周期MIPS——硬布线控制器/指令存储器必须使用ROM.png)

#### **（4.3）syscall 被误当作普通 R 型指令**

故障现象3：在控制器设计中，如果简单使用 `R_TYPE` 表示所有 `OP=000000` 的指令，并令 `RegWrite = R_TYPE OR ADDI OR LW`，则 `syscall` 也会被当作普通 R 型指令处理，可能造成停机时仍然写寄存器，或者停机控制不稳定。

原因分析：`syscall` 的 `OP` 字段同样为 `000000`，因此它确实满足 R 型指令的大类判断。但 `syscall` 的功能不是 ALU 运算，也不应该写回寄存器，而是用于结束程序执行。如果只用 `OP` 字段判断 R 型指令，就无法区分 `add`、`slt` 和 `syscall`。

解决方案：在译码时同时判断 `OP` 和 `Func`。将普通 R 型运算指令拆分为 `ADD = R_TYPE AND Func(100000)`、`SLT = R_TYPE AND Func(101010)`，将停机指令定义为 `SysCall = R_TYPE AND Func(001100)`。寄存器写使能采用 `RegWrite = ADD OR SLT OR ADDI OR LW`，停机信号采用 `Halt = SysCall`。这样 `syscall` 不再触发寄存器写入，而是单独控制 CPU 停机。

#### **（4.4）测试分析总结**

通过对 `addi`、`sw` 以及完整排序程序的测试可以看出，单周期 CPU 的关键数据通路均能正常工作。`addi` 测试验证了立即数符号扩展、ALU 加法和写回 `rt` 的路径；`sw` 测试验证了基址加偏移地址计算和数据存储器写入路径；完整 `sort.hex` 测试则综合验证了 R 型运算、立即数运算、访存、条件分支和停机指令的协同执行。调试过程中暴露的问题主要集中在控制信号译码、线路误连和指令存储器选型上，解决这些问题后，单周期 MIPS CPU 能够正确执行实验要求的 8 条核心指令。
